Como se construye un sitio con las skills de SitioHoy

Este documento explica que skills intervienen, cuando se cargan, que archivos generan, que tecnologias se usan y que controles evitan errores en diseño, Supabase, pagos, envios, QA y deploy.

Lectura rapida

El flujo esta organizado como una linea de produccion: briefing, direccion visual, scaffold, base de datos, modulos por plan, QA, launch y auditoria.

Flujo General

El proceso esta pensado para que la IA no improvise: cada etapa produce artefactos concretos y deja gates de cierre antes de avanzar.

Como se Componen y Cargan las Skills

Una skill no es solo un prompt. Es un paquete local con instrucciones, referencias, scripts, assets y templates que la IA lee de manera progresiva.

Estructura interna

skills/sitio-hoy-*/
├── SKILL.md                  # frontmatter + workflow principal
├── references/               # contratos, checklists o docs de apoyo
├── scripts/                  # automatizaciones ejecutables
├── assets/                   # HTML, templates, formularios, snippets
└── data/                     # indices, templates curados, componentes

Carga progresiva

La IA primero lee el `SKILL.md`. Solo abre referencias o scripts cuando el flujo lo pide. Eso evita cargar todo el repo en contexto y reduce errores por saturacion.

1. ActivacionEl usuario invoca la skill o el caso de uso coincide.
2. FrontmatterLa IA identifica nombre, descripcion y proposito.
3. WorkflowEjecuta pasos en orden y respeta gates.
4. ReferenciasCarga solo los archivos necesarios para el modulo.

Regla clave

El flujo principal no carga todas las skills a la vez. `sitio-hoy` orquesta, `project-director` reduce contexto y cada modulo carga su pack minimo.

Tecnologias y Librerias

El stack busca velocidad, SEO, seguridad multitenant, ISR on-demand y buen mantenimiento. Estas decisiones son parte del contrato tecnico de las skills.

Next.js 15+ App Router

Framework principal para sitios publicos, rutas server-side, metadata y deploy simple en Vercel.

  • Server Components por defecto.
  • Server Actions para mutaciones.
  • No se usa Pages Router.

ISR on-demand

Permite servir paginas rapidas y revalidar solo lo que cambia. Es clave para catalogos y tenant config.

  • `unstable_cache` + `revalidateTag(tag, 'default')`.
  • Tags por tenant para evitar colisiones.
  • `/api/revalidate` usa `tenants.revalidation_secret`.
  • Nunca `revalidatePath('/')`, `dynamic = force-dynamic` ni `revalidate = N` en catalogo.

Supabase

PostgreSQL, Auth, Storage y RLS multitenant en una instancia compartida.

  • Todo cambio se aplica con Supabase CLI.
  • RLS separa datos por `tenant_id`.
  • Triggers SQL `pg_net` llaman `/api/revalidate` leyendo `url` y `revalidation_secret` desde `tenants`.
  • Service role solo en server, nunca publico.
  • Tablas extendidas: `product_attributes`, `product_attribute_values`, `crm_webhook_config`.
  • Columnas clave en tenants: `smtp_user`, `smtp_pass`, `whatsapp`, `vercel_project_id`, `suspended_at`.
  • Columnas clave en products: `is_sale`, `sale_price`, `position`, `subcategory_id`.
  • Columnas CRM en contact_messages: `subject`, `assigned_user_id`, `labels`, `last_message_at`.

MercadoPago Bricks

Checkout para planes Emprendimiento y Empresa.

  • Preferencia creada server-side.
  • Totales recalculados en server.
  • Webhook con idempotencia y eventos en DB.

Envios

Emprendimiento usa zonas fijas. Empresa puede usar Envia.com o Correo Argentino por tenant.

  • Productos fisicos requieren `weight_grams`.
  • Credenciales por cliente en `tenants`.
  • Smoke tests por proveedor.

SMTP / nodemailer

Email transaccional via Hostinger SMTP (puerto 465/SSL) con nodemailer. Reemplaza Resend desde mayo 2026.

  • Credenciales `smtp_user` y `smtp_pass` por tenant en tabla `tenants`.
  • Templates HTML inline en `lib/email/templates.ts`.
  • Prueba de inbox antes de go-live.
  • No se marca OK si no se recibio realmente.

Umami

Analytics liviano y respetuoso de privacidad.

  • Pageviews y eventos ecommerce.
  • Plan Empresa puede requerir eventos avanzados.

UI y validacion

Librerias de soporte para experiencia, formularios y estado.

  • `lucide-react` para iconos.
  • `zod`, `react-hook-form`, `@hookform/resolvers`.
  • `zustand` para carrito.
  • `browser-image-compression` para assets del cliente.

QA tooling

Automatizaciones para detectar errores antes de mostrar o publicar.

  • Playwright para visual audit y E2E con screenshots.
  • `sitiohoy:e2e` prueba home, menu mobile, catalogo, producto, carrito, checkout y contacto.
  • Lighthouse si existe en el proyecto.
  • Secret scan, preflight, smoke tests y audit.

Gates de Calidad

Los gates evitan que el agente avance mareado o entregue un sitio con errores de diseño, integraciones o auditoria.

QA automatico

  • `npm run sitiohoy:preflight` valida brief, plan, Supabase CLI, integraciones y datos minimos.
  • `npm run build` bloquea errores TypeScript y Next.js.
  • `npm run sitiohoy:validate` revisa reglas SitioHoy.
  • `npm run sitiohoy:secret-scan` bloquea secretos commiteables.
  • `npm run sitiohoy:visual-audit` saca screenshots y detecta errores responsive.
  • `SITE_URL=http://localhost:3000 npm run sitiohoy:e2e` simula un usuario real en 375, 768, 1280 y 1920 px.

QA manual obligatorio

  • Revisar screenshots 375, 390, 768, 1280 y 1920 px.
  • Probar pagos, webhooks, emails y envios con credenciales reales cuando corresponda.
  • Confirmar que los productos sin fotos usen imagenes Unsplash relacionadas.
  • Verificar que el tracking tenga fechas Argentina `-03:00`, checks y decisiones.
  • Probar compra real, formulario de contacto, emails, envios, seguimiento y rutas 404/500 antes de produccion.
Regla de diseño: un modulo visual no se cierra si hay overflow horizontal, textos cortados, imagenes rotas, hero generico, tap targets chicos o score visual menor a 8/10. Cada sitio debe tener una direccion visual propia: paleta, composicion, tipografias y animaciones adaptadas al cliente, no una repeticion del proyecto anterior.
Regla pre-deploy: antes de publicar se limpian `console.log` y debug temporales, se corre secret scan, se revisan Client Components para evitar filtraciones y el reporte QA debe listar pendientes de produccion.
Aprendizaje continuo: cada bug corregido durante una pagina o modulo se registra en `.sitiohoy/errores-corregidos.md` con causa, solucion y recomendacion para actualizar skills.
Patterns incorporados del primer deploy: `withCache` helper para saltear cache en dev, WhatsApp desde `tenants.whatsapp` (no hardcodeado), `getTenantConfigFresh` sin cache para webhooks, ISR trigger obligatorio para tabla tenants, double-tag en queries de detalle, `refreshCartPrices` para sincronizar precios, Unsplash en `remotePatterns`, MercadoPago Payment Brick (no Wallet), email templates HTML inline, repo publico para Vercel Hobby, seed sin `compare_at_price` falso, `getCategoriesWithImages` dinamico y paso 1.5 en deploy para registrar `vercel_project_id`.

Instalacion e IAs Aceptadas

El instalador copia las skills al formato que usa cada IA y agrega el contexto base con credenciales Supabase.

Comando de importacion

Desde la carpeta del proyecto donde se quiere instalar el contexto:

curl -fsSL https://raw.githubusercontent.com/Sitio-Hoy-Tech/sitiohoy-skills/main/bootstrap.sh -o /tmp/sitiohoy.sh && bash /tmp/sitiohoy.sh

El bootstrap permite elegir version, descarga el repo temporalmente, ejecuta `install.sh` y no deja la copia completa en la computadora.

Posibles Mejoras del Flujo

Estas mejoras no son bloqueantes, pero ayudan a escalar calidad, auditoria y consistencia.

1. Generador automatico de este HTML
Crear un script que lea `SKILL.md`, planes, checklists y scripts para regenerar esta documentacion y evitar desactualizaciones manuales.
2. Dashboard de auditoria por proyecto
Convertir `proyecto-tracking.json`, `visual-report.json`, `static-report.json` y `audit-report.json` en una vista HTML por cliente.
3. QA visual con comparacion historica
Guardar screenshots por modulo y comparar cambios para detectar regresiones visuales antes de deploy.
4. Smoke tests reales por sandbox
Preparar fixtures para MercadoPago, Envia, Correo Argentino y SMTP con credenciales sandbox y resultados auditables.
5. Validador de brief mas estricto
Bloquear antes de scaffold si faltan datos claves de negocio, fotos, productos, peso, envios, contacto, dominio o referencias visuales.

Como Mantener este Documento

Cada cambio de proceso debe reflejarse aca para que el repo siga siendo entendible y auditable.

Actualizar cuando cambie

  • Una skill o su `SKILL.md`.
  • El orden de modulos de cualquier plan.
  • Un script de QA, audit, launch o tracking.
  • Una integracion: MercadoPago, SMTP/nodemailer, Envia, Correo Argentino, Umami o Supabase.
  • Una tecnologia base o libreria obligatoria.
  • Una regla transversal de diseno, E2E, seguridad pre-deploy o registro de errores corregidos.

Checklist de actualizacion

  • Editar `docs/sitiohoy-skills.html`.
  • Actualizar `README.md` si cambia el uso o la ubicacion de docs.
  • Correr `node scripts/test-skills.mjs`.
  • Correr `git diff --check`.
  • Revisar que el HTML abra localmente sin assets externos.
